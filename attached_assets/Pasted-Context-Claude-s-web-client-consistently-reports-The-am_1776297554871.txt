Context
Claude's web client consistently reports "The ambient-clinical-intelligence server is
consistently returning session terminated errors" while the skills companion (port 8002)
and ingestion (port 8003) servers remain stable. The OAuth layer was recently migrated
from in-memory Map to a PostgreSQL store (commit c95a3a6), and json_response=True, stateless=True was added to all three servers' mcp.run() calls (commit 7abb377).
Despite these fixes, the clinical server keeps tearing down sessions, forcing Claude
to fall back to the skills server.
Investigation surfaced three concrete problems, any of which can produce the reported
symptom:

OAuth tables are referenced but never created. replit-app/lib/oauth-store.ts
INSERTs / SELECTs oauth_clients, oauth_auth_codes, oauth_access_tokens, yet
no CREATE TABLE DDL exists anywhere in the repo (server/migrations/*,
mcp-server/db/schema.sql, server/deliberation/migrations/*). The moment a
cached Bearer token expires or Claude re-registers, every OAuth call raises
Postgres 42P01 relation does not exist and the flow collapses.
Lazy DB pool init on a heavy server. server/mcp_server.py is 4267 lines with
deliberation + guardrails imports. _get_db_pool() (server/mcp_server.py:677-686)
and AuditMiddleware._get_recorder() (shared/audit_middleware.py:73-88) both
init on the first tool call. The skills server has the same pattern but is
small; clinical's cold-path latency is what differs — long enough to exceed
the MCP client handshake/call window.
Clinical pool is under-provisioned relative to skills. clinical: min_size=1, max_size=5, no command_timeout; skills (mcp-server/db/connection.py:15-23):
min_size=2, max_size=10, command_timeout=30. One slow query against a missing
table can wedge the single warm connection.

Goal: make the clinical MCP server behave the same way the skills server does under
Claude Web — zero "session terminated" errors, stable reconnects, successful auth
refresh.
Root cause ranking

OAuth schema never created — latent; surfaces on token refresh or re-register.
Clinical-server cold-start latency — first tool call after a connect does
DB-pool + ensure_table work synchronously; MCP client reports the session as
dead before the response lands.
Under-provisioned pool + missing command_timeout on clinical server amplifies #2.

2. Auto-run the migration inside oauth-store.ts
Keep the footprint small — gate pool creation on a one-shot DDL promise so every
/register, /authorize, /token and /api/mcp/... bearer check guarantees the
tables exist. Edit replit-app/lib/oauth-store.ts:51-58:
tslet _pool: pg.Pool | null = null;
let _initPromise: Promise<void> | null = null;

async function ensureSchema(pool: pg.Pool): Promise<void> {
  const sql = await (await import("fs/promises"))
    .readFile(new URL("../db/migrations/001_oauth.sql", import.meta.url), "utf8");
  await pool.query(sql);
}

function getPool(): pg.Pool {
  if (!_pool) {
    _pool = new pg.Pool({ connectionString: process.env.DATABASE_URL });
    _initPromise = ensureSchema(_pool).catch((e) => {
      console.error("oauth schema init failed", e);
    });
  }
  return _pool;
}

async function withReadyPool<T>(fn: (p: pg.Pool) => Promise<T>): Promise<T> {
  const pool = getPool();
  if (_initPromise) await _initPromise;
  return fn(pool);
}
Then replace every await getPool().query(...) in this file with
await withReadyPool((p) => p.query(...)).
