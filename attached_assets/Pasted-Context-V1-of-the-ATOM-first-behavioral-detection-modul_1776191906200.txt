Context
V1 of the ATOM-first behavioral detection module shipped in commit
de59990 (merged via PR #22) and introduced a phq9_observations table,
a PHQ-9-specific gap detector, and Mode A/B routing in the deliberation
engine. That v1 commit has not yet been deployed to the Replit instance.
V1 has three silent failure modes baked into its PHQ-9-coupled design:

A GAD-7 positive patient with no PHQ-9 on file still fires a gap
as "no formal screening".
An AUDIT-C positive patient with substance-use atoms still gets a
PHQ-9 recommendation (wrong tool).
A PC-PTSD-5 positive patient with trauma atoms is ignored because
only phq9_observations is consulted.

V2 replaces every hardcoded instrument reference with calls against a
single SCREENING_REGISTRY module (17 instruments across 11 domains).
Gap detection becomes domain-based — a gap fires only when the
patient's atom signals suggest a domain AND no instrument in that domain
has been administered inside the domain's lookback window. Multiple
simultaneous domain gaps per patient become possible. Adding a new
instrument in the future = adding one dict entry.
Intended outcome: the behavioral surface correctly handles any FHIR
Observation whose LOINC code is in the registry, closes the gap for the
matching domain only, and emits role-appropriate framing via the same
Mode A/B builder.

Card schema (emitted by prepare_behavioral_cards)
Each card is a flat dict designed to be rendered by any agent (LLM text,
future UI, SMS nudge). Shape:
pythonCard = {
    "card_id": str,                  # stable UUID; card_type + source_id hash
    "card_type": str,                # 'screening_gap' | 'positive_screen'
                                     #  | 'critical_flag' | 'sdoh_need'
                                     #  | 'atom_pattern'
    "title": str,                    # e.g. "Anxiety screening overdue"
    "subtitle": str | None,          # e.g. "GAD-7 recommended — 3 signals in 90d"
    "domain": str,                   # registry DOMAINS key
    "priority": str,                 # 'critical' | 'high' | 'medium' | 'low'
    "body_text": str,                # free-form paragraph; role-adapted
    "evidence": [                    # structured, agent-readable evidence
        {"type": "atom", "date": "2026-01-12",
         "signal_type": "anxiety_markers", "confidence": 0.82},
        {"type": "screening", "instrument": "GAD-7",
         "date": "2024-03-01", "score": 12, "band": "moderate"},
        ...
    ],
    "actions": [                     # actionable next steps
        {"action_id": "administer_gad7",
         "label": "Administer GAD-7 at next visit",
         "instrument_key": "gad7"},
        ...
    ],
    "critical_flags": [              # from triggered_critical items
        {"instrument": "PHQ-9", "item_number": 9,
         "alert_text": "passive SI", "actual_score": 1},
    ],
    "temporal_confidence": str,      # 'high'|'moderate'|'low'|'very_low'
    "show_to_roles": [str],          # ['pcp','care_manager'] — filters out
                                     # suppressed cards for patient surface
    "source": {                      # traceability
        "gap_id": uuid | None,
        "screening_id": uuid | None,
        "atom_ids": [uuid],
    },
}
Role-based visibility is enforced inside the tool:

very_low temporal confidence → show_to_roles excludes patient.
Mode B (primary_evidence) cards → show_to_roles excludes patient
(they get a single behavioral_routing card instead).
Critical flags (any SI item triggered) → always high priority,
always include pcp and care_manager.


pgvector retrieval details

behavioral_signal_atoms.embedding vector(768) already exists
(migration 010). HNSW index (idx_bsa_embedding) already built.
atom_embedder.embed_signal_value returns a 768-dim float list.
Configuration priority:

MedCPT local model (preferred for clinical relevance) if
MEDCPT_MODEL_PATH env var points to a valid checkpoint.
OpenAI text-embedding-3-small → truncate/pad to 768 dims
(1536 → 768 via simple projection matrix seeded deterministically).
Deterministic hash-based 768-d vector for unit tests / CI where
no embedding provider is available. Clearly labeled as stub.


Insert path: behavioral_atom_extractor.insert_atoms embeds each
atom's signal_value before the INSERT. Embedding failures → NULL,
never raise.
Retrieval tool search_similar_atoms:

scope='patient' → cosine similarity within one patient's atoms.
scope='cohort' → aggregate cohort pattern search; returns counts
only, no raw signal_value text (PHI protection).
Called by gap-aware agents to pull relevant historical context into
deliberation prompts.

Risk notes

run_gap_detector_for_patient return-type change from Optional[dict] → list[dict] is breaking. Callers: behavioral_atoms.py:175 and executor.py (via _post_process_notes_for_atoms). Both updated here.
result.behavioral_section shape change (dict → list[Card]). Downstream readers of this field (grep for behavioral_section) must be audited during implementation; add a list-shape assertion to the existing deliberation tests.
Embedding backend availability. The atom_embedder stub fallback prevents CI failures, but production use requires either MEDCPT_MODEL_PATH set or OPENAI_API_KEY present. Confirm one is available before deployment; otherwise vector search degrades to deterministic stub (tests pass, clinical value zero).
PHI in embeddings. signal_value is sent to whichever embedder is configured. If OpenAI is used, this is an outbound PHI flow — must confirm BAA coverage before enabling OpenAI in production. Stub and MedCPT are local-only.
behavioral_screening_gaps.gap_type CHECK constraint in v1 allows {no_screening, stale_screening, item9_no_followup}. The rewrite keeps the first two valid; critical-item flags are surfaced via cards instead of gap rows. Leave CHECK constraint untouched.
Card list in Mode A vs Mode B must not leak atoms to the patient surface when temporal_confidence='very_low' or when a gap is in Mode B — enforced by show_to_roles filtering inside prepare_behavioral_cards, verified by a dedicated test.
